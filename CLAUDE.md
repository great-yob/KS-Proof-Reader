## Project overview

KS-Proof Reader — Korean-language proofreading tool for HWP (한/글) documents. It uses a modern 3-stage pipeline (Local NIKL Dictionary Screening → Google Gemini AI → Dictionary Re-validation) and writes corrections back into the HWP file via COM automation. Code, comments, and UI strings are predominantly Korean.

## Application Architecture

- **v5 (current)** — PySide6 native desktop app. Entry point: `main.py`. UI in `ui/`, engine in `core/`. This is the only codebase; the legacy v4 FastAPI + React webapp (`_archive_v4/`) was removed on 2026-06-17.

## Building for distribution

```powershell
.\.venv64\Scripts\python.exe build_dist.py --clean      # → dist/KS-Proof Reader/ (~485MB)
```

[build_dist.py](build_dist.py) (PyInstaller `--onedir`; `--onefile` is forbidden — the 160MB `stdict.db` would be re-extracted to temp on every launch). It:
1. Reads the three API keys from `config.ini`/env, Fernet-encrypts them into a generated `core/_org_keys.py`, bundles it, and **deletes it in a `finally` block** so no plaintext survives in the repo. `ConfigLoader._load_org_keys()` decrypts at runtime.

⚠ **The repo and its releases are PUBLIC, and the decryption key ships in the same file as the ciphertext — so the embedded keys are extractable by anyone.** This was an accepted trade-off (2026-07-22): the three keys are free-tier, read-only, rate-limited and individually revocable (Gemini free tier — no billing attached; 우리말샘/온용어 — free dictionary lookups), and a public repo is what makes the unauthenticated auto-updater work at all. **The rule that follows: only ever embed keys of that character.** Never add a key with billing attached, write access, or access to customer/personal data to `collect_keys()` — route those through a server proxy instead. Rotate the embedded keys if abuse shows up in the quota dashboards.

⚠ **Public repo ⇒ no customer-identifying content may be committed.** `교정샘플/` is gitignored, and measurement tables in `docs/` refer to manuscripts as 실파일A/B/C (same convention as the goldset's `실파일06/24`). Keep it that way when adding new measurements.
2. Excludes secrets/customer data from the bundle (`key.txt`, `config.ini`, `교정샘플/`, `korean-ambiguity-data/`, caches) and verifies afterwards that none leaked — **never add a rule that includes the repo root wholesale**.
3. Excludes `torch`/`transformers`/`tokenizers`/`huggingface_hub` — dead weight from the **removed KoGEC engine**, still present in `.venv64` but imported by nothing (960MB → 485MB). Unexclude only if a verified local model is ever reintroduced.

**App and data ship separately** — they change at wildly different rates, so one build emits three zips into `dist/release/`:

| Artifact | Size | When |
|---|---|---|
| `…-{ver}-full.zip` | 223MB | first install |
| `…-{ver}-app.zip` | **93MB** | code changes (frequent) |
| `…-data-{YYYY.MM}.zip` | 131MB | dictionary updates (rare) |

Data (`stdict.db`, `eomun.db`, `userdict.db`, **`kiwipiepy_model/`**) is staged **next to the EXE in `data/`**, never inside `_internal/` — bundling it there would make it unreplaceable. `verify()` fails the build if it finds `_internal/data/stdict.db`. The kiwi model is excluded from PyInstaller (`--exclude-module kiwipiepy_model`) and passed to `Kiwi(model_path=…)` instead; `kiwipiepy` never imports the model package, so this is safe.

⚠ **Never call `Kiwi(model_path=…)` without validating the path first.** A missing or mismatched model raises a catchable Python exception and *then* takes the process down with a native heap corruption (`0xC0000374`) — `try/except` does not save you. `datapaths.kiwi_model_ok()` is the guard: it checks the required model files exist **and** that the model's minor version matches the installed `kiwipiepy` (the model is version-locked to it — lib 0.23.2 ↔ model 0.23.0). `core/morph._get_kiwi()` consults it and, when frozen with no valid model, refuses to construct `Kiwi` at all. A mismatch therefore degrades to "morphology off" (graceful) instead of crashing — but that silently removes lemma restoration, the spacing backstop and the personal-name guard, so `build_dist.verify()` also fails the build on mismatch. **kiwipiepy is a dependency upgrade, not periodic maintenance**: bump it only when worthwhile, gate on `run_goldset.py`, and if the *minor* changes you must ship the data package too.

**Two version numbers, deliberately independent** — top-level [version.py](version.py): `APP_VERSION` (semver) and `DATA_VERSION` (`YYYY.MM`). The sidebar shows both on two lines. ⚠ The *authoritative* data version is `stdict.db`'s `meta.data_version` via `nikl_dict.data_version()` — `version.py` only records what a build packaged, and data updates independently. UI and updater both prefer the DB value. Release tags: `v1.0.0` (app), `data-2026.07` (data), `stdict-2026.07` (CI's partial DB).

[core/updater.py](core/updater.py) polls GitHub Releases on both channels. Data installs into `user_dir()/data` (staged → validated → swapped; datapaths ranks it first, so a restart is enough). App updates need a helper `.bat` because a running EXE can't overwrite itself. ⚠ `install_data` **refuses to run in dev** unless given an explicit `dest` — in dev `user_dir()` is the repo root, so it would clobber `data/` including the 1.8GB exports. Checking is automatic; **installing is never automatic** — a forced restart mid-proofread would destroy the user's work.

**Path resolution is centralized in [datapaths.py](datapaths.py)** — a deliberately dependency-free top-level module (importing `core` would drag in `google.genai` and the HWP bridge just to look up a word). It resolves data as user dir → EXE-side → bundle, so an updater-installed copy wins. Anything written at runtime goes to `cache_dir()` (`%LOCALAPPDATA%\KS-Proof Reader` when frozen) — the install folder and `_internal/` can be read-only. `ConfigLoader._save()` swallows `OSError` so a read-only install degrades to "settings don't persist" instead of crashing the theme toggle; when frozen, `config.ini` is **read** from both the EXE folder (deployer override) and the user folder, but **written** only to the user folder.

## Dictionary data maintenance

`data/stdict.db` (~1.64M words) has **two independently-updated halves**, and they are maintained by *different* scripts on purpose. Never rebuild one in a way that clobbers the other.

| Half | Source | Script | Cadence |
|---|---|---|---|
| `source='stdict'` (표준국어대사전, ~437K) | `spellcheck-ko/korean-dict-nikl` **`stdict/`** | CI builds `stdict_part.db` ([build_stdict_part.py](build_stdict_part.py)) → [update_stdict.py](update_stdict.py) merges | semiannual, automated |
| `source='opendict'` (우리말샘, ~1.21M) | official 전체 내려받기 export in `data/전체 내려받기_우리말샘_json_YYYYMMDD/` | [update_opendict.py](update_opendict.py) | manual, whenever the user downloads a new export |

우리말샘 can't be automated — its official export sits behind a login. The GitHub mirror *can* be, which is why CI only handles 표준국어대사전. [setup_dict.py](setup_dict.py) still does a full from-scratch build of both (`REPO_NIKL` — all three dictionaries live in that one repo; the standalone `-stdict`/`-krdict` repos have been dead since 2019).

⚠ **`norm_map` (12,750 rows, built by [build_norm_map.py](build_norm_map.py) from the 우리말샘 export) lives in the same DB file and is easy to destroy.** `update_opendict.py` builds a fresh temp DB and swaps it in, which silently dropped `norm_map` — killing every 규범표기 correction (컨텐츠→콘텐츠) with no error at all. It now rebuilds `norm_map` from the same export and gates on a table-loss invariant. `update_stdict.py` avoids the class of bug entirely by writing in-place inside a transaction. Both print an invariant check; **run `eval/ai_goldset/run_goldset.py` after any DB update** — that's what caught it.

Cadence rationale (measured 2026-07-22): upstream mirror refreshes irregularly (~semiannual); the 7-year-stale stdict source cost **0 headwords** of coverage because the fresh 우리말샘 covered all 391 gaps; and the residual gap is already handled live by the two API fallbacks. Monthly/quarterly data releases would ship 131MB for near-zero gain.

## Architecture you must understand before editing

### The 64-bit ↔ 32-bit HWP bridge (critical)

Hancom HWP's COM server (`HWPFrame.HwpObject`) is **32-bit only**, but the v5 app runs as **64-bit** PySide6. Direct `win32com.client.Dispatch` from the main process will fail.

The workaround in [core/hwp_editor.py](core/hwp_editor.py):
1. `HwpEditor` (64-bit, in-process) spawns a 32-bit Python subprocess running [core/hwp_bridge_worker.py](core/hwp_bridge_worker.py).
2. Communication is line-delimited JSON over stdin/stdout. Commands: `open`, `get_text`, `apply`, `save_as`, `close`, `quit`.
3. 32-bit Python is discovered via `PYTHON32_PATH` env var, then a list of hardcoded candidate paths (`C:\Users\user9\AppData\Local\Programs\Python\Python311-32\...` etc.), then `shutil.which`.

When changing HWP behavior, edit both sides of the bridge and keep the JSON protocol in sync. Don't try to "simplify" by removing the subprocess — it's load-bearing.

**`HwpEditor` is a factory, not a class.** [core/__init__.py](core/__init__.py) exposes `HwpEditor(file_path, ...)` which returns one of two backends with the same interface:
- Default (all extensions, incl. `.hwpx`) → the COM bridge above (`_HwpComEditor`). Required because it's the **only** backend that can apply the red-text color marking on corrected spans.
- `KS_HWP_BACKEND=direct` + `.hwpx` → [core/hwpx_editor.py](core/hwpx_editor.py) `HwpxEditor`, which edits the HWPX ZIP/OWPML XML directly (no Hancom process) but applies **text only, no color**. For fast bulk hwpx processing.

### Engine pipeline (v5)

Defined in [ui/workers/proofreading_worker.py](ui/workers/proofreading_worker.py), executed on a `QThread`:

1. **Extract** — `HwpEditor.open()` → `get_text()` (via the bridge/backend above).
2. **Local Dictionary Screening (1st Pass)** — `KoreanDictValidator` (top-level [nikl_dict.py](nikl_dict.py), *not* under `core/`) queries `data/stdict.db` to extract suspicious words (unregistered words, neologisms, dialects). [core/morph.py](core/morph.py) uses kiwipiepy to restore inflected surface forms to their dictionary lemma so genuine conjugations aren't flagged as unregistered (graceful no-op if kiwipiepy is missing). Two **cached online fallbacks** then prune false "미등재" flags, in order: [nikl_api.py](nikl_api.py) (우리말샘, serial — its lock spans the network call) → [onterm_api.py](onterm_api.py) (**온용어**, 국립국어원 전문용어 120만, parallel `ThreadPoolExecutor(4)`). Both gate on `is_likely_typo` and cache permanently in `data/api_cache.db` (separate tables `cache` / `onterm_cache` — never merge them). ⚠ 온용어 contains non-standard spellings, so `is_registered_onterm` carries a **norm_map/spelling_pairs veto guard** (`_norm_pair_components`) — see [docs/onterm-integration-design.md](docs/onterm-integration-design.md).
3. **Gemini AI** — generation is built via [core/correction_engine.py](core/correction_engine.py) `build_engine()` → `GeminiEngine`, a thin adapter over [core/gemini_checker.py](core/gemini_checker.py) `GeminiChecker` (the role-separation seam — generation is the swappable part). It uses a 30-year expert persona ([core/prompts.py](core/prompts.py)) plus the local dictionary context. API calls are chunked (`AI_CHUNK_TYPO=6000`, `AI_CHUNK_POLISH=4000`) and throttled (`AI_CALL_DELAY=4.1s`) for the 15 RPM free tier. Model: `gemini-3.5-flash-lite`, with an automatic fallback chain (`_MODEL_CANDIDATES` in `gemini_checker.py`) ordered by **free-tier RPD** — `3.5-flash-lite` (15 RPM/500 RPD) → `3.1-flash-lite` (15/500) → `2.5-flash-lite` (10/20) → `3.5-flash` (5/20). The chain advances on two distinct triggers: model unavailable (404/deprecated/permission) **and quota exhaustion** — `_is_quota_error` (narrower than `_is_transient_error`, so 5xx/timeouts don't burn a model) adds the model to `self._quota_blocked` and switches **immediately, with no backoff**, since free-tier quota is per-model. Blocked models are skipped for the rest of the process. Only when every candidate is exhausted does it fall through to `_RETRY_DELAYS` backoff.
4. **Merge** — `CorrectionMerger.merge()` deduplicates and sorts suggestions.
5. **Consistency post-pass** — [core/consistency_pass.py](core/consistency_pass.py) `enforce_consistency()` unifies AI per-chunk nondeterminism: Case A (조사 변형, high confidence) and Case 4 (부분 매칭 전파, low confidence, dictionary-guarded). `reconcile_variant_confidence()` follows. ⚠ The old Case B (edit-distance-1 whole-word substitution) is permanently removed — see file header.
6. **Dictionary Re-validation (3rd Pass)** — The AI's suggestions are cross-checked against the local DB. Non-standard terms are assigned `confidence="low"`.
7. **Apply** — `ApplyWorker` sends accepted corrections back through the bridge, and `output/errata_generator.py` emits an Excel errata file.

The `Correction` dataclass and highlighting colors (Windows COLORREF) live in [core/models.py](core/models.py). All API/chunk/throttle constants are in that same file — change tuning there, not at call sites.

### Config and API key resolution

[core/config_loader.py](core/config_loader.py) `ConfigLoader.get_gemini_key()` resolution order:
1. `GEMINI_API_KEY` env var
2. `[API] GEMINI_API_KEY` in `config.ini` (next to the EXE, or repo root in dev)
3. Encrypted org key from the build-injected `core/_org_keys.py` (see "Building for distribution").

All three keys resolve identically (env → `config.ini [API]` → encrypted org key): `get_gemini_key()`, `get_nikl_key()` (`NIKL_API_KEY`, 우리말샘), `get_onterm_key()` (`ONTERM_API_KEY`, 온용어). 우리말샘 and 온용어 are **different services — their keys are not interchangeable** (verified: the 우리말샘 key returns `returnCode 100` on 온용어). A missing Gemini key disables the app's core function; a missing dictionary key silently disables just that fallback.

### UI flow (v5) — 단일 워크스페이스 셸

[ui/main_window.py](ui/main_window.py) is a **persistent single-workspace shell**, not a wizard. The root layout is **horizontal**: a full-height left `Sidebar` ([ui/widgets/sidebar.py](ui/widgets/sidebar.py) — brand wordmark on top + embedded `StepperRail` nav + `© kim daekyung`/version footer at the bottom) and, to its right, a **main column** holding `AppHeader` (top — a **custom title bar**: theme-toggle + new-file on the left, window minimize + close on the right; the logo lives in the sidebar) / center row (`stage` + `ActivityPanel`) / `StatusFooter` (bottom). The window is **frameless** (`Qt.FramelessWindowHint`, no native title bar); `MainWindow.nativeEvent` handles Win32 `WM_NCHITTEST` to give edge-resize on all sides and turns the top `_DRAG_ZONE_H` strip (minus interactive buttons) into the move/snap region via `HTCAPTION`. The header and footer are therefore confined to the main column (separated from the sidebar). `StepperRail` exposes the 4 macro steps 설정/분석/검토/완료 with per-step result subtitles; built with `embedded=True` it drops its own width/chrome so the sidebar owns the background+border. `MainWindow.rail` is an alias for `sidebar.rail` so existing call sites are unchanged. Regions are built once and never destroyed; only the center `QStackedWidget` (`stage`) swaps among four panels in `ui/widgets/`: `SetupPanel` (upload+options merged) / `RunningPanel` / `ReviewPanel` / `ResultPanel`. `_show_phase()` keeps rail/footer/stage in sync; review is skipped when `options["auto_apply"]` is true (confidence=="low" auto-rejected). Two QThread workers (`ProofreadingWorker`, `ApplyWorker`) are owned by the window; their signals fan out to the persistent components. `_cleanup_worker` requests stop and waits up to 5s before discarding — keep that pattern when adding new workers.

**Design system**: [ui/styles/theme.py](ui/styles/theme.py) holds `LIGHT`/`DARK` token palettes, scales (`SPACE`/`RADIUS`/`FONT`), and `build_qss(palette)`. Style **only** via global QSS objectName/property selectors (e.g. `variant="primary"`, `role="card"`, `state="active"`) + `restyle(widget)` after a property change — avoid per-instance `setStyleSheet` in persistent widgets so dark-mode toggle reskins everything via `apply_theme(app, mode)`. Directly-painted/HTML widgets (`ToggleSwitch`, review `QTextBrowser`, icons, activity log) implement `refresh_theme()`; `MainWindow._refresh_all_themes()` walks the tree and calls it on every descendant after a toggle. Reuse primitives from [ui/widgets/components.py](ui/widgets/components.py) (`make_button`, `card`, `section_card`, `badge`, `chip`, `label`). Theme mode persists via `ConfigLoader.get_theme()/set_theme()` ([APP] section).

**Bundled assets** (under `assets/`, resolved by [ui/styles/assets.py](ui/styles/assets.py) for dev + frozen): `assets/fonts/` Pretendard **TTF** (the `static/alternative` TrueType-hinted build — OTF/CFF rendered blurry & vertically clipped on Windows/Qt; loaded by [ui/styles/fonts.py](ui/styles/fonts.py) `load_fonts()` in `main.py`; primary UI font, set via single-family `font-family: "Pretendard"` since Qt QSS ignores comma fallback lists; the app font uses `QFont.PreferNoHinting` because Windows' default hinting grid-snaps and drops thin top strokes e.g. ㅎ), `assets/icons/` Lucide SVGs (ISC) rendered/recolored to QIcon/QPixmap by [ui/styles/icons.py](ui/styles/icons.py) — use `IconButton`/`IconLabel` or `make_button(..., icon="name")`, never emoji glyphs. **Build note**: Nuitka/PyInstaller must bundle `assets/` and the `PySide6.QtSvg` + `qsvg` image-format plugin.

**Review panel** ([ui/widgets/review_panel.py](ui/widgets/review_panel.py)): cards are **per-occurrence** — a word repeated N times in the manuscript yields N cards (each tagged `반복 k/N`), ordered by document position; clicking a card scrolls the preview to that occurrence (`<a name>` anchors + `scrollToAnchor`). Cards default to **pending** (white); each occurrence has its own accept/reject (split ✓/✕ solid icon buttons, card tinted green/red by `status` property). **반복 일괄 처리**: a user's accept/reject on any card auto-propagates to the group's other untouched repeat cards (`occ["auto"]`); flipping an auto-set card to the opposite raises a confirm popup and, if confirmed, flips **only that card** (`occ["by_user"]` — never overwritten by later propagation). **용어 일관성 통일** (`_unify_dialog`): 띄어쓰기 일관성 카드(marked `Correction.consistency_flip` — set **only** by worker [7] `find_compound_spacing_consistency` consumption; `consistency_pass` variants explicitly unset it; never enable for norm corrections) is **not** a per-item verdict but a document-wide 표기 choice, so `_set_status` routes it to a confirm popup **before** the repeat/auto branch (accept → "문서 전체를 '{corrected}' 표기로 통일", reject → "…'{original}'…"). Confirm ⇒ the whole group moves together **and** the opposite direction is settled in the same step: reject runs `_apply_flip` (reject A→B group + synthesize/accept reverse B→A), accept runs `_apply_unify_forward` (accept A→B group + reject an existing reverse so the two can't cancel out). Symmetric — selecting on the reverse card returns to the original direction; no reverse card is ever synthesized for non-`consistency_flip` cards (that would de-standardize norm corrections). Consequence: consistency cards have **no per-occurrence partial** accept/reject. The old "반대 표기로 통일" link button was removed 2026-07-21 (users rejected without noticing it, leaving both 표기 mixed). Both paths preserve other cards' per-occurrence states by re-running only sort/`_mark_stem_boundary_skips`/`_resolve_overlaps`/`_derive_all` (**never** rebuild occurrences — that wipes partial decisions), and both are dispatched via `QTimer.singleShot(0, …)` after the popup closes (dialog teardown first, else a ghost white window; `_set_status` must `return` immediately since `_rebuild_cards` deletes the card widget it was passed). The popup is `LightConfirmDialog` — a frameless **always-light** modal built on `build_light_card()`/`LIGHT_CARD_QSS`, the same surface the **toast** (`_toast`, 반복 일괄 처리 알림) uses, so both read as one system: head title + divider + body, drawn in-widget so the OS dark title bar can't leak in, every visual property re-declared under `#lcd*` objectName selectors to beat the global dark QSS, and wordWrap labels given a measured `setMinimumWidth` (their sizeHint collapses to ~230px and folds a one-liner into 3 lines). Both are theme-independent — do **not** add `refresh_theme()` to them. **Card list loading is measured-tuned** (constants at the top of `ReviewPanel`): a 50-card batch cost ~433ms of frozen UI, and since each batch carries a ~30ms whole-list layout pass, smaller batches only multiply the total — so the fix isn't batch size. Instead: first batch `_INITIAL_CARDS` (screen-filling only), the rest by an **idle prefetch timer** that runs only after `_IDLE_QUIET_MS` of no interaction (`_note_interaction()` is called from scroll/accept-reject/`_scroll_to`) and **parks itself once `_PREFETCH_AHEAD` viewports of cards are buffered below the fold** — never preload everything, because insert cost scales with how many cards are already alive. `_on_scroll` keeps an 80% safety net (`_SCROLL_CHUNK`). `_rebuild_cards` swaps the whole scroll content (`_new_scroll_content`) instead of detaching cards one by one (309ms → 64ms for 200 cards, and no widget ever becomes a top-level window, which also retires the old ghost-window workaround) — ⚠ `QScrollArea.setWidget()` destroys the old widget **synchronously**, so clear `self._cards` first. Never call `_emit_counts()` from `_load_more_cards`: counts derive from `_occ`, not from loaded cards. ⚠ **A freshly created card's `pos()` is not final for several layout passes** (measured y: 0 → 877 → 2953); reading it after a single `singleShot(0)` lands mid-settle and scrolls to the wrong place, and forcing `activate()`/`adjustSize()` does not fix it. Both `_scroll_to` and `_restore_card_anchor` therefore go through `_place_card(occ, offset, smooth)`, which re-measures until the position stops changing. Any rebuild-triggering action must bracket itself with `_capture_card_anchor(occ)` / `_restore_card_anchor(..., on_done=)` to keep the acted-on card at the same viewport offset, and must hide the rebuild: `_do_unify` loads cards **up to the anchor synchronously** (deferring that leaks a frame showing only the initial batch) and wraps the whole sequence in `_card_scroll.setUpdatesEnabled(False)`, re-enabled from `on_done` plus a time-based backstop so the panel can never stay frozen. Likewise `_refresh_preview(keep_scroll=True)` is the default because `setHtml` resets QTextBrowser scroll to 0 — accepting one card used to fling the 원문/교정문 panes back to the top; only `load()` passes `keep_scroll=False`. It suspends `_scroll_sync` around the swap and restores each scrollbar twice (immediately + next tick, since the document's scroll range isn't final until it has laid out). To make the previews *follow* an action instead, call `_scroll_previews_to(cid)` (previews only — leaves the card list where it is) from a `QTimer.singleShot(0)` so it runs **after** that restore; `_do_unify` does this to match what `_scroll_to` gives normal cards, and passes `active_occ=` into `_apply_flip`/`_apply_unify_forward` so `_mark_active_occ` can re-resolve the occurrence index after the re-sort without a second render. ⚠ the panel's correction list is a sorted **copy**, so `main_window._start_apply` re-syncs `self._corrections = review_panel.get_corrections()` before applying — without it panel-added reverse corrections never reach ApplyWorker/errata. **Partial reject is real**: `_derive()` sets the unique `Correction.status` (accepted if any occurrence accepted) and `skip_occurrences` = occurrence indices (document order, 0-based) NOT accepted; this flows `apply_worker` → `HwpEditor.apply_corrections` → bridge `apply()`, whose `RepeatFind` loop skips those `count` indices (and disables the AllReplace fallback when a skip set is present). `counts_changed`/`get_counts()` report **occurrence-level** counts. **Card width is fixed to the column** (no h-scrollbar): the 원본 label gets ZWSP soft-break points (`_soft_breakable`, default size policy — a custom `Ignored` policy broke heightForWidth), the 교정 field is a `GrowingTextEdit` (wrap-anywhere QTextEdit that grows vertically, `QLineEdit`-compatible `text()`/`editingFinished`; height = doc height + measured chrome, **never** `frameWidth()` which lies under QSS borders), and cards are vertically `Maximum` so leftover list space goes to the trailing stretch, not into the cards.

**Result = completion dashboard**: [ui/widgets/result_panel.py](ui/widgets/result_panel.py) renders a report-style dashboard — header (파일명+완료 시각+`StampBadge` 완료/검수 도장+정오표·교정본 폴더 액션) / pipeline hero (01 분석 N자 → 02 교정 제안 사전·규칙/AI 분해 → 03 적용 N건·본문 M곳·적용률, terms: 건=항목·곳=등장) / 상세 분석 (제안 소스 도넛·적용률 게이지·오류 유형 바) / 실패항목(인라인 사유, no dialog) / 교정 진행 로그, all inside a vertical QScrollArea. Numbers are the heroes: `AnimatedNumberLabel` count-up (OutExpo + 착지 팝, final-width reserved so layout doesn't shift; font set via per-instance stylesheet because global QSS `QWidget{font-size:13px}` overrides `setFont`). `show_result(result, log, corrections=, char_count=, file_name=)` — main_window passes the extras at apply-done; the errata-regeneration call passes only (result, log) and the panel reuses stored extras (timestamp kept via result-dict identity). `refresh_theme()` re-renders statically (no re-animation). Chart widgets read `current_palette()` at paint time; only bundled Lucide icons exist (no pie-chart/clock/file-spreadsheet — use table/circle-check/list-checks etc.). The progress log there is the *filtered* proofreading narrative from `ActivityPanel.get_proofreading_log()` (excludes HWP-bridge plumbing via `_PLUMBING_KW`); the right `ActivityPanel` keeps the full raw log (incl. `[Worker stderr]`) for debugging. There is no settings dialog — the header has only theme-toggle + new-file (the HWP-version-registration `settings_dialog.py` and its header button were removed 2026-06-17; the Gemini API key comes from env/config.ini/embedded org key).

## Conventions

- Korean docstrings/comments are the norm. Keep new code consistent.
- `core/` is GUI-agnostic — never import `PySide6` from `core/`. UI threading lives in `ui/workers/`.
- Long-running checkers accept a `threading.Event` named `stop_event` and a `logger` callable; honor both.
- There is no automated test suite. Verify changes by running the app (`.\.venv64\Scripts\python.exe main.py`) against a real `.hwp`/`.hwpx`; `core/` modules are GUI-agnostic and can be exercised directly from a Python REPL.
