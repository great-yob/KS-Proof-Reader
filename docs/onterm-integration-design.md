# 온용어(국립국어원 전문 분야 용어지식 플랫폼) 통합 설계

작성 2026-07-22 · 형제 문서: [proofreading-architecture.md](proofreading-architecture.md), [userdict-layer-architecture.md](userdict-layer-architecture.md)

> 전제: **사전 = 항상-on 검증/가드 인프라, 생성 = 교체형 AI**.
> 온용어는 **사전 층(억제 인프라)** 으로만 들어온다. 생성·치환 권한은 주지 않는다.

**결론 먼저** — 실측 결과 효과는 **거짓 검수 카드 약 8% 감소**(실문서 3건, 153건 중 12건).
→ **API 폴백(Phase 1)만 도입한다. 120만 건 벌크 DB(Phase 2)는 권장하지 않는다** — 효과
상한이 같은데 배포물 수십 MB·라이선스 필터·수동 수집 절차를 떠안게 된다. 근거는 §5·§6.

## 1. 온용어란 무엇인가

| 항목 | 내용 |
|---|---|
| 정체 | 국립국어원 **전문 분야 용어지식 플랫폼** (`https://kli.korean.go.kr/term`) |
| 이력 | 2024-09 시범 → **2025-05-15 정식 개통** |
| 규모 | **약 120만 용어 / 53개 용어집 / 36개 국가·공공기관** (2025-05 개통 기준) |

| 분류 체계 | **온용어 분류 체계: 대분류 5 / 중분류 42** (`category_main`/`category_sub`) |
| 완비율(rate) | 필수 정보 4종(**분야·정의문·사용 예시·원어**) 충족 개수 = ★0~4 |
| 용어 간 관계 | 표준 전문 용어 / 표준화 대상어 / **다듬은 말 / 다듬을 말** / 일치어 / 동의어 / 참고어 |
| 일치어 | 형태·개념이 같지만 서로 다른 용어집에 실린 용어를 하나로 묶은 것 |
| 표제어 표기 | NIKL 관례 — `^` = 구 경계(`스마트^그리드`), `-` = 음절 경계 |
| 인증키 | 16진수 32자리, **유효기간 2년**(만료 시 재발급). 우리말샘 키와 **별개** |

> "국가 과학기술 표준 분류체계 22/277"). **현행은 소개 페이지·API 기준 5/42**,
> 규모는 개통 보도자료 기준 120만/53/36. 도움말을 근거로 코딩하지 말 것.

## 2. 오픈 API 실측 명세 (2026-07-22, 유효 키로 검증)

```
GET https://kli.korean.go.kr/term/api/search.do
    ?key=<32자리>&apiSearchWord=<검색어>&start=1&num=30
```

### ⚠ 공식 문서와 실제가 다른 지점 — 여기가 구현 함정 전부다

| # | 실측 | 공식 문서 | 대응 |
|---|---|---|---|
| 1 | `start`/`num` 생략 시 **JSON이 아니라 HTML 페이지** 반환 | "선택" | **항상 명시**. 생략 금지 |
| 2 | `num=2000` → **1894건 전부 반환**(1.3MB, 0.8s). 상한 없음 | `10~100` | 상한 신뢰 말 것 |
| 3 | `start=501` 이상 → **0건**(total과 무관) | `1~1000` | 페이지네이션은 **~500**까지. 다만 `num`이 무제한이라 `start=1`+큰 `num`이면 불필요 |
| 4 | 성공 시 `return_object`는 **list**, 오류·무결과 시 **str** | 명시 없음 | **타입 분기 필수** (`isinstance`) |
| 5 | 성공 `returnCode`는 **정수 `1`**, 오류는 **문자열 `"100"`** | integer | **`str()` 씌워 비교** |
| 6 | 결과 0건 → `return_object = "검색 결과가 없습니다."` (`total` 없음) | 명시 없음 | **확정 미등재(False)** — 오류(None)와 구별해야 함 |
| 7 | 무효 키 → `returnCode "100" "시스템 에러"` (020 아님) | 020 Unregistered | **키 오류와 요청 오류를 코드로 구별 불가** → `!= "1"` 이면 전부 None(확인 불가) |
| 8 | 키 누락 → `"000" "key 값이 없습니다."` | 000 = System Error | 〃 |
| 9 | 검색이 **부분 일치(substring)** — `전력` → `열기전력`도 매칭 | 명시 없음 | `method=exact`가 **없다** → 응답 `word`를 정규화해 **직접 정확일치 판정 필수** |
| 10 | 공백 무시 — `스마트그리드`·`스마트 그리드` 동일 결과 | 명시 없음 | 조회 전 공백 제거 무해 |
| 11 | 조사 붙으면 무결과 — `스마트 그리드를` → 0건 | 명시 없음 | **조사 제거 후 base로 조회 필수** |
| 12 | 미문서화 필드: `INSTT_ENG_NM`, `GLOSSARY_ENG_NM`, `DFNTXT_ORIGIN`, `TRGT_WORD_TY` | – | 의존 금지 |
| 13 | `kr_gvrn_lcns_ty`가 **빈 문자열**인 항목 존재 = 우리말샘 출처(CC BY-SA) | integer | 빈 값 처리 필요 |
| 14 | 본문에 HTML 엔티티 이중 이스케이프(`&lt;strong&gt;`, `&#44;`) | – | 정의문·용례 쓸 거면 언이스케이프 필요(우린 안 씀) |
| 15 | `relate_word`에 **관계 유형이 들어 있다** — `"(다듬은 말, 허위생성)"`, `"(동의어, 초매개변수), (일치어, 철통인증)"` | "관련 용어명" | **API만으로 다듬을 말 판별 가능** |

- **일일 호출 제한 = 5만 건**(인증키 신청 페이지 명시). 인증키는 **사용자당 1개**만 발급.
  문서당 수십 건이므로 **사실상 무제한** — 호출 예산은 설계 제약이 아니다.
- 정렬 기본 `wt`(정확도) — 정확일치가 최상위로 온다(`전력` 검색 시 `전력`이 1위). `num=30`이면 충분.
- 상세 조회 API(`terms.korean.go.kr/api/view`)는 표준국어대사전 view와 동일 스키마.
  뜻풀이·용례·발음은 교정 판정에 안 쓰므로 **우리 앱에 불필요**.

### 벌크 내려받기(웹 UI)

`용어 목록 내려받기` — 엑셀/워드/JSON/XML, 항목 선택(분야·원어·정의문·사용예시·관련용어·
대역어·참고사항·완비율·**공공누리 유형**). `자세히 찾기`의 필터(공공누리 유형·관계 유형·
완비율·용어집)를 상속한다. **⚠ 회원 로그인 필수** → 무인 자동화 곤란.
공공데이터포털에 온용어 전량 파일데이터는 **없다**(우리말샘만 존재).

## 3. 라이선스

**용어마다 공공누리 유형이 다르다**(`kr_gvrn_lcns_ty`).

| 유형 | 상업적 이용 | 변형 | 우리 앱 |
|---|---|---|---|
| **제1유형** | 가능 | 가능 | ✅ 유일하게 안전 |
| 제2유형 | **금지** | 가능 | ❌ |
| 제3유형 | 가능 | **금지** | ❌ (DB 파싱·정규화 = 변형) |
| 제4유형 | **금지** | **금지** | ❌ |

- 전 유형 **출처 표시 의무**. 우리말샘 출처분(`lcns=""`)은 **CC BY-SA 2.0 KR**(동일조건변경허락).
- 실측상 **주요 용어집이 4유형이다** — TTA 《정보 통신 용어 사전》, 《표준 용어 사전》,
  《국방 과학 기술 용어 사전》 모두 `lcns=4`. 벌크로 가면 **이들을 통째로 버려야 한다.**
- 전례: 바른 중의성 데이터셋(CC BY-NC)은 사내 상업 맥락이라 레포 반입 금지, 외부 clone만
  허용(memory: `det-layer-phase-d-and-bareun`). 온용어도 같은 잣대.

**→ 핵심 함의**: API 폴백은 응답을 **재배포하지 않는다**. 캐시에 남는 건
`(단어, boolean, 조회시각)`뿐이라 2차적 저작물이 아니고 로컬이라 배포도 아니다.
**라이선스 부담이 사실상 0.** 반면 벌크는 표제어 집합을 배포물에 넣으므로 1유형 필터가 필수다.

파이프라인 [2] 스크리닝을 그대로 재현해, `is_likely_typo` 게이트를 통과한
(= 실제로 AI 의심어/검수 카드가 되는) 어휘를 온용어에 조회했다.

| 문서 | 분량 | dict_flags | 조회 대상 | **온용어 구제** | 효과 |
|---|---|---|---|---|---|
| 실파일A(사회복지 연구용역) | 69,636자 | 73 | 45 | 3 | 6.7% |
| 실파일B(IT·법률 연구용역) | 166,640자 | 61 | 34 | 6 | **17.6%** |
| 실파일C(스타트업·AI 보고서) | 268,088자 | 152 | 74 | 3 | 4.1% |
| **합계** | 504K자 | 286 | **153** | **12** | **7.8%** |

> 실측 원고는 고객 자료라 파일명을 익명화한다(골드셋의 `실파일06/24` 관례와 동일).
> 원본은 레포에 없다 — `교정샘플/`은 .gitignore 대상.

- 구제된 어휘: 할루시네이션·하이퍼파라미터·제로트러스트·보안사고·로보틱스·포커스·퀄리티 등
- **진짜 오탈자는 하나도 구제되지 않았다** — 상담채녈·캐릴터·괌심·페밀리·훗가이도현 전부 잔존.
  즉 **recall 손실 0**(이 표본에서). 억제 방향이 안전하다는 실증.
- 잔존 대다수는 외래어 음차(리크루팅·리텐션·풀스택·에쿼티)·인명·기관명 → **온용어로도 못 줄인다.**
- 지연: 직렬 조회 평균 **0.53s/건**, 문서당 34~74건 → **18~39초**. (완화책은 §5-3)

**해석**: 도메인 편차가 크다(4~18%). 우리 사용자의 실제 원고는 보고서·연구용역이 많아
**한 자릿수 후반이 기대값**이다. 작지만 방향이 100% 억제라 손해가 없다.

## 5. Phase 1 설계 — API 폴백

### 5-1. 파일별 변경

**① `core/config_loader.py`** — `get_nikl_key()` 패턴 그대로 `get_onterm_key()` 추가.
`ONTERM_API_KEY` 환경변수 → `config.ini [API] ONTERM_API_KEY` → `""`.

**② `onterm_api.py` (루트, `nikl_api.py`의 형제) — 신설**

```python
def available() -> bool
def exists_online(word) -> Optional[bool]   # True 등재 / False 미등재 / None 확인불가
```

`nikl_api.py`를 본뜨되 §2 함정 15종을 전부 반영한다. 특히:

| 항목 | nikl_api (우리말샘) | onterm_api (온용어) |
|---|---|---|
| 엔드포인트 | `opendict.korean.go.kr/api/search` | `kli.korean.go.kr/term/api/search.do` |
| 정확일치 | `method=exact` 있음 | **없음 → `word` 정규화 직접 비교 필수** (`^`·`-` 제거) |
| 필수 파라미터 | `key,q,req_type,advanced,method` | `key,apiSearchWord,start,num` (**start/num 생략 시 HTML**) |
| 성공 판정 | `total > 0` | `isinstance(return_object, list)` **and** `str(returnCode)=="1"` |
| 무결과 | `total == 0` | `return_object == "검색 결과가 없습니다."` (str) |
| 캐시 테이블 | `cache` | **`onterm_cache` — 별도 테이블** (우리말샘 결과와 절대 섞지 말 것) |

**③ `nikl_dict.py`** — `is_registered_online()` 옆에 `is_registered_onterm(word)`를
동일 시그니처로 추가(조사 제거 `_JOSA_RE` 재사용).

**④ `ui/workers/proofreading_worker.py` [2]** — 우리말샘 폴백 **다음 단**에 삽입:

```
dict_flags
  → (1단) 우리말샘 API : is_likely_typo(w) and is_registered_online(w) → 제외
  → (2단) 온용어  API : is_likely_typo(w) and is_registered_onterm(w)  → 제외   ← 신규
  → 남은 것만 AI 의심어 + 검수 카드로
```

순서 근거: 우리말샘이 일반어를 먼저 걷어내 온용어 호출 수(=일일 제한 소모)를 줄인다.
로그: `→ 온용어 API: 전문용어 N건 확인·제외 (거짓 검수 방지)`.
`022` 수신 시 **즉시 영구 비활성**(`_disabled=True`) + 로그 1줄, 재시도 루프 금지.

### 5-2. ⚠ 필수 가드 — norm_map 거부권 (실측 근거)

**온용어에는 비표준 표기가 등재되어 있다.** 실측:

```
'컨텐츠' 조회 → '디지털^컨텐츠' (국방 과학 기술 용어 사전, lcns=4)
```

`컨텐츠`는 우리 `norm_map`의 규범표기 교정 키다(`컨텐츠 → 콘텐츠`). 그런데 `디지털컨텐츠`는
norm_map 키가 **아니어서**, 온용어가 이 어절을 통째로 화이트리스트에 넣으면 **비표준 표기를
정상으로 승인**하게 된다.

**가드**: 온용어 등재 판정이 나와도, 해당 어절이 `norm_map`·`spelling_pairs`·`userdict_pairs`의
키를 **성분으로 포함**하면 화이트리스트를 적용하지 않는다(온용어가 규범 판단을 이길 수 없다).
→ 이 가드는 Phase 1 구현에 **반드시 함께** 들어간다.

### 5-3. 지연 완화

직렬 0.53s × 최대 74건 = 39초는 체감된다. 두 가지 중 택일:

- **(권장) 병렬 조회** — 독립 GET이므로 `ThreadPoolExecutor(max_workers=4)`로 묶으면
  39초 → 약 10초. 4워커는 예의 범위(현재도 0.12s 간격 삽입 중).
- **(대안) 캐시 선행** — 첫 실행만 느리고 `onterm_cache` 히트 후엔 0초. 그대로 두기.

우리말샘 폴백도 같은 직렬 구조라 **같이 병렬화하면 기존 지연도 함께 준다**(별건 작업).

### 5-4. 하지 않는 것 (명시적 금지)

| 금지 | 이유 |
|---|---|
| **다듬은 말/순화어 자동 치환** | 외래어 순화 제외 가드(`파라미터가→매개변수가` 차단), `[K]` 유의어 치환, `[U]` 병기 명칭 치환 전례. 출판 교정에서 순화어 강제 = 저자 의도 침해 |
| **표준 전문 용어 페어 자동 적용** | 도메인이 맞아야 성립하는데 문서 도메인을 우리는 모른다 |
| **정의문·용례의 AI 프롬프트 주입** | KAGEC/RAG 교훈 — 프롬프트 주입 방식 전면 폐기됨 |
| **온용어 등재를 "교정 근거"로 사용** | 여러 기관 용어집의 통합체라 비표준 표기가 섞여 있다(§5-2 실증). 등재는 "고치지 말라"는 근거일 뿐 "이렇게 고치라"는 근거가 아니다 |
| **`is_likely_typo` 게이트 없이 전 어휘 조회** | 일일 제한 즉시 소진 + 스크리닝 지연 폭증 |

### 5-5. 출처 표시

## 6. Phase 2(벌크 DB)를 권장하지 않는 이유

| | 근거 |
|---|---|
| **효과 상한이 같다** | 화이트리스트가 걷어낼 수 있는 건 `dict_flags` 중 `is_likely_typo` 생존분뿐 — 경로가 API든 로컬 DB든 **동일한 153건이 모수**다. 8%가 천장 |
| **주요 용어집이 4유형** | TTA 정보통신·표준 용어 사전·국방 과학 기술 전부 `lcns=4` → 1유형 필터를 걸면 정작 전문용어 핵심이 빠진다 |
| **수집이 수작업** | 로그인 필수 + 웹 UI 조작. 1회 내려받기 상한도 미공개 |
| **배포물 비대** | `stdict.db`가 이미 160MB. 여기 수십 MB 추가 |
| **갱신 부채** | 온용어는 계속 증축 중("공개 예정 용어 수" 별도 표시). 스냅샷은 즉시 낡는다 |
| **API 캐시가 사실상 같은 것을 준다** | `onterm_cache`는 쓸수록 자라는 로컬 사전이다 — 실제 원고에 나온 어휘만 골라 담으므로 **120만 건보다 밀도가 높다** |

**재검토 조건**: 일일 제한은 5만 건으로 확인돼 **더는 재검토 사유가 아니다**(문서당 수십 건).
남은 사유는 **오프라인 환경 배포 요구**뿐이다. 그 경우에도 **1유형 + 다듬을말 제외 +
완비율 하한** 필터를 건 축소판만 만든다.

## 7. Phase 3(선택) — 띄어쓰기 근거

온용어 표제어의 `^`는 **권위 있는 구 경계 정보**다(`스마트^그리드`, `공공^기관`).
현재 띄어쓰기 일관성은 **문서 내 다수결**이 원칙이고, 메모리 ⑲에 따라 **동률일 때만**
"사전 등재면 붙임 / 미등재면 띄어쓰기"로 방향을 정한다.

→ `^` 정보를 **동률 상황의 tie-breaker 근거로만** 쓴다. 다수결이 이미 방향을 정했으면
**개입 금지**. 산출은 저신뢰 검수 카드까지.

⚠ 다수결 로직을 건드리므로 **골드셋 Phase D/E 게이트 통과가 선결**이며,
메모리 `occurrence-count-stem-boundary`의 "붙임형/띄어쓴형/패널 3자 카운트 기준 통일"
제약을 반드시 함께 볼 것. **Phase 1 안정화 전에는 착수하지 않는다.**

## 8. 회귀 게이트

온용어는 **억제 방향 전용**이므로 게이트가 단순하다.

- `eval/run_goldset.py` 기준 **finder/카드 발화 수가 줄기만 해야 한다. 늘면 FAIL.**
  (Phase E 델타 게이트와 동일 철학)
- 줄어든 항목 중 **진짜 오탈자가 섞이지 않았는지** 반드시 눈으로 확인 — §5-2의
  비표준 표기 등재 문제 때문. 현재 표본(153건)에서는 recall 손실 0이었다.
- `norm_map` 거부권 가드(§5-2)의 단위 케이스로 `디지털컨텐츠`를 골드셋에 넣는다.

## 9. 한 줄 요약

> 온용어는 **새 교정 엔진이 아니라 과교정 억제용 전문용어 화이트리스트**다.
> 실측 효과는 거짓 검수 카드 **약 8% 감소**(recall 손실 0). **API 폴백 + 영구 캐시로만**
> 도입하고 120만 벌크 DB는 만들지 않는다. 다듬은 말·표준 전문 용어의 자동 치환은 영구 금지,
> `norm_map` 키를 포함한 어절에는 화이트리스트를 적용하지 않는다.
